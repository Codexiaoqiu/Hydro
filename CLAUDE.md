# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this project is

**Hydro** — a high-performance, AGPLv3 online judge platform. The codebase is a Yarn 4 monorepo of TypeScript packages that compile via the project's own on-the-fly TS loader at runtime (see `framework/register`). Hydro is the successor to `vijos/vj4`.

- Live service: https://hydro.ac
- Docs: https://hydro.js.org
- AGPLv3 license with additional terms in `LICENSE` (do not remove copyright, must rename/renumber modified versions, no use of author trademarks without permission).

## Workspace layout

Yarn workspaces (see `package.json`):

- `packages/hydrooj` — **core server**: handlers, models, services, CLI, plugin loader, worker process, MongoDB layer.
- `packages/hydrojudge` — **judge daemon**: sandbox client, judge flows (default / interactive / objective / communication / hack / submit_answer / generate), and the `bin` entrypoint that polls host backends.
- `packages/common` — shared types (`STATUS`, `PERM`, `PRIV`, `STATUS_TEXTS`, `subtask`, `cases`, etc.) consumed by both server and judge.
- `packages/ui-default` — backend handlers + **Nunjucks/React/webpack-bundled** frontend (the user-facing UI).
- `packages/ui-next` — Vite + React 19 SPA with a slot-based plugin system and a CSS-variable design system (dark-first, Geist-inspired light alternative). Renders via the `next` renderer with priority 100 / `asFallback: true`, so it coexists with ui-default. Currently ships `homepage` and `problem_main` pages styled via `src/styles/tokens.css` + component-local `*.module.css` (Button, Card, Chip, Eyebrow, LangTabs, TagCloud, Avatar, Ring, Trend, IDEFrame, SamplePair, Article, CtaCard, Menu, ContestList, Author, TopNav, BrandMark, NavLink, LangPill, ThemeToggle). Theme is persisted via `localStorage('hydro.theme')` and resolved SSR-safely by `THEME_INIT_SCRIPT` in `src/theme/theme-init.ts`.
- `packages/<addon>` — official addons: `blog`, `center`, `elastic`, `fps-importer`, `geoip`, `import-hoj`, `import-qduoj`, `login-with-github|google|oidc`, `migrate`, `onlyoffice`, `onsite-toolkit`, `prom-client`, `scoreboard-xcpcio`, `sonic`, `telegram`, `vjudge`, `a11y`, `components`.
- `framework/framework` — `Handler`, `ConnectionHandler`, `Router`, `WebService`, `@param` decorator, validator, error classes, schema types — **the abstraction every package depends on**.
- `framework/register` — `@hydrooj/register`: a Node `require.extensions` shim that on-the-fly transpiles `.ts/.tsx/.jsx/.js` to CommonJS via esbuild (and reads inlined source maps).
- `framework/utils` — `@hydrooj/utils`: shared utilities (`fs`, `Time`, `Logger`, `parseMemoryMB`, `sysinfo`, etc.).
- `framework/eslint-config` — the shared ESLint + oxlint configuration (rebuilt via `yarn oxlint`).
- `examples/plugins/*` — example third-party plugins.
- `plugins/`, `modules/` — empty containers; a `nop.ts` placeholder is generated by `build/prepare.js` on first install (the workspace is gitignored-empty until someone populates it).
- `install/` — the bundled single-file installer (built with `yarn build:installer`).

## Common commands

Node **>=22** is required (`init.ts` hard-fails on older versions).

```bash
# install (also runs build/prepare.js to wire symlinks and tsconfigs)
yarn

# build all backend packages (tsc -b)
yarn build

# production UI build (writes public/ bundles + __bundleInfo diff)
yarn build:ui:production

# UI dev server with hot reload (port 8000, proxies 2333)
yarn build:ui:dev
# or HTTPS variant
yarn build:ui:dev:https

# rebuild just the iconfont (svgtofont) — must run before any UI build
yarn build:ui:iconfont

# lint with auto-fix
yarn lint
# lint in CI (no warnings allowed)
yarn lint:ci
# fast oxlint pass
yarn oxlint

# run tests (spawns the full server via mongodb-memory-server, runs /test/main.ts)
yarn test

# autocannon benchmark — produces benchmark.json
yarn benchmark

# regenerate the self-installing installer script (gzip + base64 of an esbuild bundle)
yarn build:installer

# start a dev instance with hot-reload of all plugins
yarn debug
# or
yarn start

# interactive CLI (db / addon / diagnosis / install / patch)
yarn hydrooj <command>

# ui-next component tests (vitest + happy-dom + testing-library)
yarn workspace @hydrooj/ui-next test          # single-shot (22 tests across 5 components)
yarn workspace @hydrooj/ui-next test:watch    # watch mode

# ui-next visual regression (Playwright, requires `playwright install chromium` once)
yarn workspace @hydrooj/ui-next build                 # build production bundle first
yarn workspace @hydrooj/ui-next test:visual            # compare against baselines
yarn workspace @hydrooj/ui-next test:visual:update     # regenerate baselines
```

The `tsconfig.json` files in each package are **generated** by `build/prepare.js`. If you see a stale tsconfig, run `node build/prepare.js` (or just `yarn`).

### Running a single test

There is currently one `.spec.ts` (`packages/common/tests/subtask.spec.ts`) and an end-to-end suite (`test/main.ts`). To run only the unit-style spec:

```bash
node -r @hydrooj/register packages/common/tests/subtask.spec.ts
```

`test/main.ts` runs as a `node:test` `describe('App')` block against a fully-spawned `hydrooj` process with `process.env.CI=true` (which switches MongoDB to `mongodb-memory-server`). To iterate on it use `yarn test`.

## Architecture

### Plugin / service model

`packages/hydrooj/src/context.ts` exposes a **`cordis` Context** extended by `ApiMixin`, the `Loader` service, `i18n`, `logger`, `timer`, and an `EventMap` (`bus.ts`). Almost every cross-cutting concern is a `cordis` `Service`:

- **boot order** (`entry/worker.ts`): `MongoService` → `SettingService` → `SystemModel` → `Loader` → `HMR` → `storage`, `worker`, `server` services → all `handler/*.ts` files (`loadDir`) → `migration` → addons → `script/` → `app/started` → `app/listen` → `app/ready`.
- **plugin loading** (`loader.ts`): `Loader.reloadPlugin(key, configScope)` `require()`s the module, resolves its `Config` / `schema` against the setting service, then `ctx.plugin(plugin, config)`s it inside a scoped Cordis fiber. It listens to `system/setting` to hot-reload when config changes.
- **addons** are detected by walking `getAddons()` (`options.ts`) plus the local `packages/<name>` directory; `global.addons[name]` maps to the absolute path.
- **DEV mode** is toggled by `--debug` to the loader or `process.env.DEV`. In DEV the loader warns the user about AGPLv3 obligations and adds `-<hash>-dirty` to the version string via `simple-git`.

### HTTP layer (`framework/framework/server.ts`)

- `Handler` and `ConnectionHandler` are the two base classes. `WebService` exposes `Route(name, path, HandlerClass, ...permPrivChecker)` and `Connection(name, path, ConnectionHandler, ...)` to register them. `register()` records both into a custom `Router` (`framework/framework/router.ts`) and into the per-handler layered middleware pipeline.
- A handler request goes through a deterministic step list (`init → __prepare → _prepare → prepare → handler/before → all|get|post|post<Operation> → handler/after → cleanup`). Plugins can intercept at any `handler/...` event via `ctx.serial` / `ctx.parallel`.
- CSRF: a POST with a `referer` header that differs from `host` is rejected (`CsrfTokenError`) unless `allowCors` is set or CORS is enabled.
- Permission (`bigint`, `PERM.PERM_*`) and privilege (`number`, `PRIV.PRIV_*`) checks are passed positionally into `Route()` and become the request's checker.
- The `@param('name', Types.X, optional?)` decorator (`decorators.ts`) builds per-method validators that run automatically before the handler body. Validation uses **schemastery** schemas.
- Renderers are pluggable: `WebService.registerRenderer(name, { render, accept, priority, output })`. Default is JSON via `serializer.ts`.

### Models (`packages/hydrooj/src/model/`)

Plain TS modules that own a single MongoDB collection (typically `coll = db.collection('name')`). Important ones:

- `builtin.ts` — re-exports `PERM`/`PRIV`/`STATUS` from `@hydrooj/common` and lists every `PERMS` entry shown in the admin UI.
- `problem.ts`, `record.ts`, `contest.ts`, `discussion.ts`, `domain.ts`, `user.ts`, `training.ts`, `homework.ts`, `token.ts`, `oplog.ts`, `opcount.ts`, `solution.ts`, `storage.ts`, `task.ts`, `blacklist.ts`, `schedule.ts`, `system.ts`.
- `setting.ts` exports the schemastery setting hierarchies: `PreferenceSetting`, `AccountSetting`, `DomainSetting`, `DomainUserSetting`, `SystemSetting`. `settings.ts` (`SettingService`) is the runtime that loads, validates, and migrates them.

### Service layer (`packages/hydrooj/src/service/`)

- `db.ts` — `MongoService`: wraps a single `MongoClient`, exposes `collection<K>(name)` (honours `prefix` and `collectionMap`). In CI it spawns `MongoMemoryServer`.
- `server.ts` — re-export of `WebService`.
- `worker.ts` — `WorkerService`: consumes `schedule` collection entries every second with a 20-minute per-task cap; supports `interval` for recurring tasks.
- `monitor.ts`, `check.ts`, `bus.ts`, `hmr.ts`, `storage.ts`, `migration.ts`, `watch.ts`.
- `layers/` — Koa middleware layered onto `WebService`.

### Judge subsystem (`packages/hydrojudge/`)

The judge is **two entry points**:

1. **`packages/hydrojudge/bin/hydrooj-judge` (daemon.ts)** — a long-running worker that consumes `hosts[*].type` (either `HydroHost` or `Vj4Host`). It polls the host's task queue (default: `hydrooj`'s `TaskModel.consume({ type: 'judge' }, ...)`), fetches test data, then runs `JudgeTask.handle()`.
2. **`hosts/builtin.ts`** — registers the judge as a Cordis plugin **inside** the hydrooj process itself (when run with `NODE_APP_INSTANCE === '0'` and `disable !== true`), wiring `TaskModel.consume({ type: 'judge' | 'generate' }, ...)` directly into the same node process. This is the standard single-host deployment.

Per-task pipeline (`task.ts` → `flow.ts` → `judge/<type>.ts`):

1. Compile (`compile.ts`) — runs the language-specific compile command from `SettingModel.langs` (declared in `packages/hydrooj/setting.yaml`).
2. Judge type dispatch (`judge/index.ts`) — picks one of `default | interactive | objective | communication | hack | submit_answer | generate`. Default `default.ts` runs `runFlow`, which iterates **subtasks** (`subtask.type ∈ {min, max, sum}`) and within each subtask runs cases concurrently with `p-queue`, aborting siblings when a `min`-type subtask is already failing or a dependency failed (`ctx.failed[i]`).
3. Each case invokes `runQueued` (`sandbox/index.ts` re-export of `sandbox.ts`), which talks over HTTP to the `go-judge` sandbox binary (installable separately; version checked by `sandbox.ts::versionCheck`). Statuses are translated via `statusMap` (e.g. `TimeLimitExceeded → STATUS_TIME_LIMIT_EXCEEDED`).
4. Checker program (`checkers.ts`) runs the user-configured checker (standard, `testlib`, `lemon`, `syzoj`, `hustoj`, etc.) over the program's stdout/stderr.
5. Results stream back through `JudgeResultCallbackContext` (`packages/hydrooj/src/handler/judge.ts`), which mutates the `record` document (`processPayload`) and pushes incremental updates over a WebSocket / SSE (`ConnectionHandler`).

The judge implements `multi-pass` (`mp.i`) for problems whose checker returns a follow-up input, and runs a single re-judge on TLE under 5 s.

### Sandbox (`packages/hydrojudge/src/sandbox/`)

`sandbox/client.ts` is the HTTP client for the `go-judge` daemon (a separate process, not part of Node). `sandbox.ts` constructs `Cmd` parameters with CPU-time scaling (`getConfig('rate')`), `copyIn`/`copyOut`/`copyOutCached`, optional `addressSpaceLimit`, `processLimit`, `stdio_size` limits, and env passthrough. `versionCheck` ensures the running `go-judge` binary meets the minimum required version.

### Frontend (`packages/ui-default/`)

- **Webpack-bundled** UI (Vijos-style). Entry: `index.ts` (which registers backend routes like `WikiHelpHandler`, `MarkdownHandler`, etc.); the actual page templates live in `pages/` and use Nunjucks. The webpack config is in `build/config/webpack`.
- `build/main.ts` runs webpack (or webpack-dev-server on port 8000 with `/ws` excluded from proxying to `localhost:2333`). With `--production` it also writes `__bundleInfo` and prints a size diff vs. last build.
- `--iconfont` regenerates `misc/.iconfont` via `svgtofont` from `misc/icons/*.svg`.
- Locales in `locales/*.yml` (e.g. `en.yml`, `zh_CN.yml`) with the substitution syntax documented in `CONTRIBUTING.md`. **YAML keys must be sorted alphabetically ascending.**

### Translation / i18n

Translations live in **per-package** `locale/<lang>.yml` files. Backend: `_('key').format(args...)` or `i18n('key').format(...)`. Frontend: `substitute(i18n('key'), args)`. Substitution is positional `{0}` or named `{name}`. Fallback chain: `ko_KR.yml → ko.yml → en.yml → key`.

### Server startup entry points

- `packages/hydrooj/bin/hydrooj` — CLI (`cac`) — dispatches `addon / db / diagnosis / install / patch` subcommands (`entry/cli.ts`) or, when no subcommand, runs `entry/worker.ts` which boots the full server in master/worker mode via `pm2`-style `NODE_APP_INSTANCE` forking.

### HMR (`packages/hydrooj/src/service/hmr.ts`)

In DEV mode, server-side plugins are hot-reloaded when their source changes. Triggered by `--watch` to the worker entry.

## Conventions worth knowing

- **Don't add info-bearing comments, docstrings, or translation strings via AI without reviewer sanity-check.** `CONTRIBUTING.md` is explicit: contributors remain responsible for submitted content; auto-submissions are not allowed; reviewers must understand every change.
- **AGPLv3 obligations** apply (see `LICENSE` and the in-loader DEV warning): you must open-source modifications under AGPLv3 and retain copyright. The loader prints a 5-second warning banner when running in DEV mode.
- **Decorators are central**: handlers use `@param`, services use `static inject = [...]` and `static Config = Schema.object({...})` (a schemastery schema). Plugins extend `Service` and listen to events via `ctx.on(...)`.
- **Error handling**: throw `HydroError` subclasses (`UserFacingError` for things the user should see, `BadRequestError`, `NotFoundError`, `ForbiddenError`, `ValidationError`, `CsrfTokenError`, `MethodNotAllowedError`, `InvalidOperationError`, `ProblemIsReferencedError`, `FileLimitExceededError`). The framework maps them to status codes and templates (`error.html` / `bsod.html`).
- **Schemas use `schemastery`**, not Zod. `Schema.object({ ... })`, `Schema.array()`, `Schema.union()`, etc. Validators in handler decorators and service configs all flow through it.
- **Status/Permission/PRIV constants** live in `@hydrooj/common` (`status.ts`, `permission.ts`). Always import from there, never duplicate.
- **Database is MongoDB** (single client, multiple collections). There is no SQL. Collection naming is configurable via `prefix` and `collectionMap` on `MongoService`.
- **Frontend imports use the `vj/*` alias** that resolves to `packages/ui-default/*` — that alias is generated by `build/prepare.js` into the relevant tsconfigs.
- **Test runner is `node:test`** (not Mocha/Jest). The e2e harness boots the full server in-process and uses `supertest.agent`.
- **No `prepare` is needed in tests** — `framework/register` transpiles TS on the fly via esbuild and decodes inlined source maps; `node -r @hydrooj/register ...` is the typical entry.

## Build artifacts and caches

- `.cache/ts-out/` — `tsc -b` outputs.
- `packages/ui-default/public/` — webpack bundles (`.map` files are deleted after each production build).
- `packages/ui-default/__bundleInfo` — last production bundle size baseline (used for the diff).
- `~/.hydro/static` — runtime-copied static asset dir assembled from every addon's `public/` (worker `entry/worker.ts`).

## Pinned runtime

`package.json` uses `yarn@4.6.0` (Corepack) and Node ≥22. The repo also ships a `flake.nix` for Nix-based installs (the loader detects `NIX_PROFILES` to wire in nix-installed plugins).
